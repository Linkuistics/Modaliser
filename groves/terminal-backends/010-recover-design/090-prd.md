# 090 — PRD

Write `docs/prd/terminal-backends.md` capturing the agreement
reached in 080. PRD is the human-facing, team-shareable face of
this planning increment; lives forever, never retired.

## Required content

- **Scope.** The 13-op surface + detection primitive. What's in,
  what's deferred (close, resize, zoom, swap, rotate; SSH).
- **Backend matrix.** Each backend's surface (full / detection-only),
  gate conditions (e.g. ghostty ≥ 1.4), prerequisites
  (`allow_remote_control` for kitty), and a one-line capability
  summary.
- **Abstraction shape.** Whatever 080 picked — mechanism + naming +
  surface split + mux-inside-host composition. Include the ADR
  references.
- **Known limitations.** "Indirect and inexact" cases per backend:
  e.g. ghostty chip-positioning, alacritty multi-window tty
  disambiguation, zellij detection workaround.
- **Migration / compatibility.** What breaks for existing configs
  (e.g. if names change from `focus-pane-left` to `focus-pane-h`).
- **Future work.** Close-pane, resize, mux-aware automatic
  context-switching, etc.

## Skip conditions

If 080 ended with "no convergence," this task instead writes a
short note in `notes/no-prd.md` recording the reason, and the node
retires without a PRD.

## After this task

The synthesis is complete. The node's last live leaf is done; the
grove loop says: promote anything still relevant from the node's
`BRIEF.md` upward to the root brief or to ADRs, then
`mv 010-recover-design/` into `groves/terminal-backends/done/`.
Then create the implementation node (likely `020-implement/`) with
per-backend implementation leaves seeded from this PRD.
